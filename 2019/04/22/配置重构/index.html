配置重构

发表于 2019-04-22 | 更新于 2019-04-23 | 分类于 基础建设 | 评论数：

本文对配置重构的重要性进行说明，对配置重构产生的价值进行梳理。

背景

原来在融汇一代的配置是以XML格式存放在程序的config目录下。大多数程序都有企业配置、银行配置、以及其他的业务相关配置。所有的银行配置都存放在BankConfig.xml中，所有企业配置存放在SubScriConfig.xml。而随着银行渠道增多，企业越来越多，业务复杂性提升，配置越来越臃肿。目前的配置已经不便于运维维护，也不再适应于微服务的架构。

重构的价值

目前融汇二代处于研发阶段，主要还是依赖于融汇一代的业务支持，而融汇直联原来的架构并未考虑到运维的可维护性，且没有很好的代码及配置规范，造成程序配置，程序部署，程序迁移的的工作量越来越大。而对于后期规划来说，无论是融汇一代还是融汇二代，都会往微服务架构方向发展。

适用于微服务架构

微服务架构具有可扩展性，独立性，原来的配置方式不再适用与微服务架构。主要原因是原来每个程序会有一份配置文件，甚至每份配置文件可能还不一样。这样的配置的方式造成服务存在数据相关性，对企业迁移造成了极大的困难，对于运维维护上造成大量的重复工作量，且极易改错或改漏配置。因为配置作为最重要的基础项之一必须优先优化。

跨平台

微服务架构，通过统一的接口调用，即可实现多平台互联互通。每个平台都有自己的技术优势，通过服务调用可以屏蔽不同平台的语言差异，环境差异。合理的利用不同平台的优势。

这里所说的平台指的是技术平台，如.net、java或其他平台等。

自动化

目前二代已经开始使用自动化部署，而一代也已经在进行自动化的建设及试用。通过配置重构以后，充分的考虑到自动化的需求。最终实现一键部署，一键升级的能力。这其中都需要极为扎实的基础框架建设。

动态迁移和扩容

在微服务的架构之上实现服务的高可用。同时还需要实现服务动态扩容和动态迁移。程序和本地配置已经剥离开来，迁移程序无需在迁移配置或缓存信息。极大的降低了运维工作量。

最终愿景

搭建运维平台，降低配置管理工作量，降低程序升级和程序部署的工作量，降低排查问题的复杂程度。

  • 配置管理：因为只保存了一份配置，因此更为容易的进行配置维护，同时根据需求拆分配置，解决目前配置全部堆积在一个配置文件所造成的一席列问题。
  • 服务管理：通过在持续集成和持续部署的基础之上，可视化进行服务管理。包括服务关闭、服务启动、服务重启、服务升级、服务回退等。同时也为服务部署、服务迁移和服务扩容提供了基础。通过“一键”部署，“一键”迁移极大的减轻当前运维人员碰到的运维痛点。
  • 日志管理：将日志汇总后，通过界面进行查看，查找。

技术可行性

  1. 配置管理可使用二代页面或重新开发，无技术难点。
  2. 服务管理，包括服务安装，服务卸载，验证windows服务动态安装卸载。
  3. 确认服务自动升级方案，包括包升级，配置升级，数据库脚本升级，无技术难点。
  4. 确认服务回滚方案，包括包回滚，配置回滚，数据库脚本回滚，无技术难点。
  5. 确认服务迁移方案，包括服务部署，配置更新，无技术难点。
  6. 确认服务扩容方案，包括服务部署，配置更新，无技术难点。

如何重构

方案一

依然以xml格式存储，将之前的配置按银行和企业将对应的配置拆分到不同文件。好处是部分配置节点依然可以使用，降低运维首次更新配置的工作量。将配置拆分到多个配置文件，更便于配置查找和修改。支持重载配置(企业配置配置覆盖公共配置)，减少公共配置量。通过SVN方式托管配置，有一个统一的配置数据源，也适用于微服务架构。

方案二

目前二代是将配置存储到数据库。可以延用二代的方案，好处是可以做到一二代配置通用，同时在初期运维可以通过数据库对数据进行增删查改。相比直接修改文本文件，更为方便一些。同时也适用于微服务架构。二代目前已经有了界面化配置，可以直接使用二代的界面化配置进行配置修改，而一代加载数据即可。
