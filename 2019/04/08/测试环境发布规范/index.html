持续集成流程

graph LR
    获取代码 --> 编码
    编码 -.-> 编写单元测试
    编写单元测试 -->上传代码
    编码 -->上传代码
    上传代码 --人工触发-->jenkins编译
    上传代码 --自动触发-->jenkins编译
    jenkins编译 --编译成功--> 执行单元测试
    jenkins编译 --编译失败--> 获取代码
    执行单元测试 --单元测试成功--> 持续部署
    执行单元测试 --单元测试失败--> 获取代码
  1. 开发人员获取代码进行编码
  2. 编码后若有需要则编写单元测试，单元测试通过后，进行自测功能。
  3. 测试通过后上传代码。
  4. 人工执行或自动触发jenkens构建。
  5. jenkins构建失败，则开发重新获取代码解决问题。
  6. jenkins构建成功，有必要则执行测试。
  7. 单元测试执行通过后开始持续部署。

持续部署流程

graph LR
    持续集成 --成功--> 备份原始文件
    备份原始文件 --> 上传文件
    上传文件 -.失败.-> 恢复备份文件
    恢复备份文件 --> 人工干预处理
    上传文件 --成功--> 修改配置
    修改配置 --> 执行数据脚本
    执行数据脚本 --> 关闭服务
    关闭服务 --> 启动服务
    持续集成 --失败--> 编码
  1. 持续集成完成后备份目标服务器对应程序的库包文件。
  2. 将jenkens编译成功后的库包文件上传到目标服务器对应的程序目录下。
  3. 人工修改配置文件。
  4. 人工执行数据库脚本。
  5. 人工重启服务

测试环境发布测试流程

针对首次在.net 直联引入CI(持续集成),CD(持续部署),现在测试环境对流程进行试验及合理化流程。

原始流程

  1. 开发编码本地测试通过打包
  2. 开发根据配置修改编写配置修改文档
  3. 开发编写数据库脚本
  4. 将库包，配置修改文档和数据库脚本都上传到SVN的冲突包。
  5. 通知测试，测试更新冲突包的dll，将源程序关闭。并复制新的库包到程序目录。
  6. 根据配置修改文档修改配置。
  7. 执行数据库脚本。
  8. 启动程序进行测试。

目前测试环境和开发环境是分开的，因此开发通过jenkins持续集成后将全量包上传致测试环境FTP。
测试人员通过开发给出的测试文档人工修改配置，通过开发给出的数据库脚本人工执行脚本。
